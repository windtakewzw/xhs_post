三种架构方案对比

小红书自动发布系统 — 20账号 · 房地产垂直 · 全自动AI内容生产

A

方案A：现有 MCP + 管理层封装

基于 xhs-mcp 或 mcp-xiaohongshu 做底层发布，上层自建：账号管理、内容流水线、定时调度

优点

  • 发布能力开箱即用，开发周期短
  • 登录/Cookie管理已有现成方案
  • 社区持续维护，适配平台变更

缺点

  • 受限于MCP的功能边界，定制困难
  • 依赖第三方维护，断更风险
  • 多账号支持需自行封装
B

方案B：Playwright 全栈自研

基于 Playwright 完全自定义浏览器自动化，自建全部功能模块

优点

  • 完全可控，可处理任意边缘场景
  • 多账号隔离天然支持（独立BrowserContext）
  • 可定制反检测策略，降低封号风险

缺点

  • 开发工作量大，需处理登录、发布全流程
  • 需自行跟进XHS前端DOM变更
  • 视频发布实现复杂度高
C

方案C：混合架构（推荐）

MCP 做常规发布通道 + Playwright 做兜底和复杂场景，上层统一调度

优点

  • 快速启动 + 长期可控，兼顾速度和灵活性
  • MCP 失效时自动降级到 Playwright
  • 可按场景选择最优通道

缺点

  • 需维护两套发布通道
  • 架构复杂度略高
  • 需要定义清晰的通道抽象接口
